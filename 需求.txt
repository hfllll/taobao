第一部分：需求分析
1.1 项目概述
1.1.1 一句话说清楚
我们要做一个面向消费者的智能在线商城系统，用户可以浏览商品、搜索商品、加入购物车、下单购买、查看订单状态，商家可以管理商品信息和处理订单，系统还能根据用户浏览和购买历史自动推荐个性化商品，解决传统购物方式选择困难、信息过载以及商家运营效率低的问题。
1.1.2 项目范围：做什么
本系统需要完成以下功能：
• 商品浏览与搜索：支持按名称、分类、价格区间等条件搜索和筛选商品；
• 购物车管理：用户可以将商品加入购物车、修改数量、删除商品；
• 订单管理：用户提交订单、查看订单状态、取消订单、申请售后；
• 用户管理：用户注册登录、个人信息管理、收货地址管理；
• 商品管理：商家录入商品、修改商品信息、上下架管理、库存管理；
• 订单处理：商家查看订单、确认发货、处理退款/退货申请；
• 智能商品推荐：系统根据用户浏览和购买历史，自动推荐可能感兴趣的商品；
• 评价系统：用户购买后可对商品进行评分和评价，帮助其他用户决策。
其中，本次项目重点关注的核心特色功能为：智能商品推荐。

1.4 核心功能详细描述（选3个最重要的）
功能1：商品搜索与浏览
项目	内容
使用者	消费者（无需登录也可浏览和搜索商品）
前置条件	用户已打开商城首页或商品列表页面
基本流程	1. 用户在搜索框中输入商品关键词 → 2. 可选择分类、价格区间、品牌等筛选条件 → 3. 可选择排序方式（综合排序/销量优先/价格升序/价格降序/评分优先）→ 4. 点击"搜索"按钮 → 5. 系统根据关键词和条件查询数据库 → 6. 系统返回商品列表，展示商品图片、名称、价格、销量、评分 → 7. 用户可点击商品进入详情页查看更多信息（规格参数、商品描述、用户评价）
预期结果	系统正确展示匹配搜索条件的商品列表，商品信息准确完整；若无匹配结果提示"未找到相关商品，试试其他关键词吧"；支持分页加载，每页20条商品。
异常情况	• 搜索关键词为空：提示"请输入搜索关键词"
• 搜索超时（>3秒）：提示"搜索超时，请稍后重试"
• 数据库连接失败：提示"服务繁忙，请稍后再试"
• 特殊字符注入：系统进行输入过滤和参数校验，防止SQL注入和XSS攻击

功能2：下单购买
项目	内容
使用者	消费者（已登录）
前置条件	消费者已登录系统；购物车中有至少1件商品；已设置默认收货地址；商品库存充足
基本流程	1. 消费者在购物车页面点击"去结算" → 2. 系统跳转订单确认页面，展示商品清单、总金额 → 3. 消费者选择或新增收货地址 → 4. 消费者选择支付方式（在线支付/货到付款）→ 5. 消费者点击"提交订单" → 6. 系统校验库存（若库存不足则提示）→ 7. 系统创建订单（状态：待付款），锁定库存 → 8. 跳转支付页面，消费者完成支付 → 9. 支付成功后订单状态更新为"待发货"，系统通知商家
预期结果	订单创建成功，库存正确扣减，消费者可在"我的订单"中查看订单详情；商家收到新订单通知。
异常情况	• 未登录：跳转到登录页面，提示"请先登录"
• 库存不足：提示"XX商品库存不足，当前仅剩X件，请调整数量"
• 收货地址为空：提示"请先添加收货地址"
• 支付失败：订单保留为"待付款"状态，15分钟内可重新支付，超时自动取消
• 并发下单（多人同时购买最后一件商品）：使用数据库悲观锁/乐观锁，确保只有一人成功
• 网络超时：提示"网络异常，请检查网络后重试"

功能3：智能商品推荐
项目	内容
使用者	消费者（已登录时个性化推荐；未登录时热门推荐）
前置条件	消费者已登录（个性化推荐需有浏览/购买记录）；未登录用户展示全站热门商品
基本流程	1. 消费者登录后进入商城首页 → 2. 系统分析消费者的浏览记录（浏览过的商品分类）和购买记录 → 3. 系统使用混合推荐策略：
   a) 基于内容的推荐：根据用户浏览/购买的商品特征（分类、品牌、价格区间）推荐相似商品
   b) 协同过滤推荐：找到与该用户偏好相似的其他用户，推荐他们购买过的商品
   c) 热门推荐：补充全站热销商品确保推荐列表多样性 → 4. 系统生成个性化推荐列表（去重、过滤已购买商品）→ 5. 在首页"猜你喜欢"板块展示推荐商品（图片、名称、价格、推荐理由标签如"与你浏览过的XX相似"）→ 6. 消费者可点击推荐商品进入详情页并购买
预期结果	首页展示8-12个个性化推荐商品；推荐商品与用户偏好高度相关；推荐列表根据用户行为实时更新；点击率相比随机展示提升30%以上。
异常情况	• 新用户无浏览/购买记录：展示全站热销商品作为冷启动推荐，附带提示"浏览更多商品后，将为您提供个性化推荐"
• 推荐算法计算超时（>2秒）：降级为缓存的热门商品列表
• 推荐结果不足8个：用热门商品和新品上架补充至最低展示数量



第二部分：系统设计
2.1 系统架构概述
2.1.1 我们选什么架构？
项目	选型	理由（为什么选它）
架构风格	☑ 前后端分离	前后端分离架构将前端（Web端/移动端）与后端业务逻辑解耦。前端负责页面渲染和用户交互，后端通过RESTful API提供数据和业务服务。优势：前后端可并行开发、独立部署；一套后端API可同时服务Web端和移动App；技术栈独立选型，团队可按专长分工。
前端框架	Vue 3 / Element Plus /tailwindCss .daisyUI .Vue 3是国内最主流的前端框架之一，生态丰富，学习成本低。Vant UI专为移动端商城设计，提供商品卡片、购物车、订单列表等现成组件。Element Plus用于商家管理后台，提供表格、表单、图表等企业级组件。
后端框架	Spring Boot + MyBatis-Plus	Spring Boot是Java生态最成熟的微服务框架，自动配置、内嵌Tomcat，开发效率高。MyBatis-Plus提供强大的CRUD封装和分页插件。事务管理能力保障订单、支付等关键业务的数据一致性。Java技术栈在国内电商领域有大量成功案例（如淘宝、京东后端均为Java技术栈）。
数据库	MySQL 8.0 + Redis + Elasticsearch	MySQL存储商品、订单、用户等核心结构化数据，支持ACID事务保障交易一致性。Redis缓存热点数据（首页商品列表、用户Session），大幅降低数据库压力，提升响应速度。Elasticsearch实现商品全文搜索，支持中文分词、拼音搜索、模糊匹配，相比MySQL LIKE查询性能提升数十倍。
AI工具	混合推荐引擎（内容推荐 + 协同过滤 + 热度排序）	采用混合推荐策略实现智能商品推荐：
1) 基于内容的推荐：提取商品特征向量（分类、品牌、价格段），计算与用户历史偏好商品的余弦相似度；
2) 基于物品的协同过滤（ItemCF）："购买此商品的用户也购买了..."，利用用户行为数据挖掘商品关联；
3) 热度加权：引入商品销量、评分、上架时间等因子对推荐结果进行重排序。混合策略解决了单一算法的冷启动和多样性不足问题，推荐效果更优。

2.3.1 模块调用关系说明

订单模块 → 商品模块	下单时需要校验商品库存，创建订单后需扣减库存	GET /api/products/{id}/stock（查询库存）
PUT /api/products/{id}/stock（扣减库存）
订单模块 → 用户模块	下单时需获取用户收货地址，验证用户身份和状态	GET /api/users/{id}/addresses（获取地址列表）

推荐模块 → 订单模块	推荐算法需要获取用户的历史购买记录来分析偏好	GET /api/orders/history?userId={id}（获取购买历史）
GET /api/orders/stats（获取全站销售统计）
推荐模块 → 商品模块	生成推荐列表后需要查询推荐商品的详细信息	GET /api/products?ids={id1,id2,...}（批量查询商品信息）
评价模块 → 订单模块	评价前需要验证用户是否已购买该商品（防止刷评）	GET /api/orders/check?userId={id}&productId={pid}（验证购买资格）
评价模块 → 商品模块	评价后需要更新商品的综合评分	PUT /api/products/{id}/rating（更新商品评分）
购物车模块 → 商品模块	加入购物车时需要查询商品信息（价格、库存、状态）	GET /api/products/{id}（查询商品详情）
